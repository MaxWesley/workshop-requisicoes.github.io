031 · Como a internet funciona

Tudo é alguém pedindo e alguém respondendo

Quando você abre um site, seu computador (o cliente) manda um pedido para outro computador (o servidor), que devolve a resposta. Igualzinho a uma carta.

  • Endereço (URL) — pra onde a carta vai. Ex.: judit.io
  • DNS — a lista telefônica que descobre o "número da casa" (o IP) a partir do nome.
  • Pacote — a carta viaja em pedacinhos e é remontada no destino.
042 · O protocolo HTTP

HTTP é a etiqueta dessa conversa

É o conjunto de regras combinado para pedir e responder. Sempre em pares: uma requisição vai, uma resposta volta.

O cliente nunca recebe nada sem pedir primeiro. Por isso "fazer uma requisição" é, literalmente, fazer um pedido seguindo essa etiqueta.

062 · Os verbos

Quatro verbos dão conta de quase tudo

GETLer / buscar. "Me mostra o que tem aí." Nunca muda nada.
POSTCriar. "Adiciona uma coisa nova." Na Judit: criar uma consulta.
PUTAtualizar. "Troca o que já existe por essa versão."
DELETEApagar. "Remove essa coisa." Na Judit: parar um monitoramento.
072 · Status

O semáforo da resposta

Toda resposta volta com um número de três dígitos dizendo o que aconteceu.

2xxDeu certo200 OK, 201 Criado. O pedido funcionou.
4xxErro de quem pediu401 sem chave, 404 não existe, 400 mal formado.
5xxErro do servidor500 algo quebrou do outro lado. Não foi você.
083 · JSON

JSON é uma ficha que a máquina lê

Pares de chave: valor, agrupados entre chaves. É assim que os dados vão e voltam nas requisições.

  • Cada informação tem um nome (a chave) e um conteúdo (o valor).
  • Vale texto, número, verdadeiro/falso e listas.
  • É o formato que a API da Judit usa pra entregar processos.
resposta.json
{
  "nome": "Maria Souza",
  "tem_processos": true,
  "quantidade": 3,
  "tribunais": ["TJSP", "TJRJ"]
}

115 · A API da Judit

Uma requisição. Todos os tribunais.

Em vez de manter robôs em cada tribunal, o cliente faz uma chamada HTTP e a Judit busca e devolve os dados prontos. Autenticação por api-key no cabeçalho.

requests

Assíncrono

Vai ao tribunal em tempo real. Resposta chega depois.

lawsuits

Síncrono

Lê o datalake da Judit. Volta em milissegundos.

tracking

Monitora

Acompanha o processo e avisa quando muda.

125 · O conceito-chave

Na hora, ou depois?

A diferença que mais gera dúvida no cliente.

SÍNCRONOEspera na linhaPede e recebe na hora, do datalake. Milissegundos.
ASSÍNCRONORecebe um protocoloVolta um request_id; o resultado chega depois, quando o tribunal responder.
136 · Passo 1 — criar

Pedindo os processos de um CPF

Um POST abre a consulta. A Judit responde na hora com um protocolo (request_id) e status pending.

Repare: a chave vai no header api-key — não é Authorization: Bearer.

POST /requests
POST https://requests.production.judit.io/requests
api-key: SUA_CHAVE

{
  "search": {
    "search_type": "cpf",
    "search_key": "999.999.999-99"
  }
}

// resposta
{ "request_id": "84b4d8f5-...", "status": "pending" }
acompanhar & obter
// 2 · já ficou pronto?
GET .../requests/84b4d8f5-...
api-key: SUA_CHAVE

→ { "status": "completed" }

// 3 · pegar o resultado
GET .../responses?page=1
api-key: SUA_CHAVE

→ { "page_data": [ /* processos */ ] }
146 · Passos 2 e 3

Acompanhar e colher o resultado

Com o protocolo em mãos, você checa o status até virar completed e então busca os dados.

  • GET /requests/{id} — "já terminou?"
  • GET /responses — colhe os processos em JSON.
  • Ou webhook: a Judit te avisa sozinha quando ficar pronto.
156 · Monitoramento

O webhook: a resposta que liga de volta

No tracking você cadastra um processo uma vez. Quando há andamento novo, a Judit chama o cliente — sem ele perguntar.

POST /tracking
{
  "recurrence": 1,
  "search": {
    "search_type": "lawsuit_cnj",
    "search_key": "1111111-04.1111.1.11.1111"
  }
}
167 · Por que isso importa

Agora você traduz o cliente

o cliente diz

"Minha requisição deu 401"

Você sabe: é a api-key faltando ou errada no cabeçalho. Resolvido em segundos.

o cliente diz

"O resultado não veio na hora"

É uma consulta assíncrona: voltou um protocolo, o dado chega depois — ou via webhook.

o cliente diz

"O webhook não chegou"

Você entende o monitoramento e sabe pra quem encaminhar com o contexto certo.

o cliente diz

"Preciso de dado em tempo real"

Você aponta o serviço certo: requests on-demand, não o datalake síncrono.
